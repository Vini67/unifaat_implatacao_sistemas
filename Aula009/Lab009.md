## Passo 1: Gerenciamento de Chaves (Key Pairs)

### O que são Key Pairs?

Key Pairs são pares de chaves criptográficas (pública/privada) usadas para autenticação SSH. A AWS mantém a chave pública, você guarda a privada.

### Criando Key Pair no Console

**1. Acesse EC2:**
- Console AWS → EC2 → Key Pairs (no menu lateral esquerdo)

**2. Create Key Pair:**
- Name: `unifaat-ads-key`
- Key pair type: **RSA** (mais compatível)
- Private key file format: **.pem** (para Linux/macOS)

**3. Download:**
- O arquivo `unifaat-ads-key.pem` será baixado automaticamente
- **🚨 GUARDE ESTE ARQUIVO COM SEGURANÇA**
- **🚨 SE PERDER, NÃO CONSEGUIRÁ ACESSAR A INSTÂNCIA**

### Configurando Permissões da Chave

```bash
# Mover chave para diretório do projeto
mv ~/Downloads/unifaat-ads-key.pem ~/ImplantacaoSistemas/Aula09/

# Alterar permissões (OBRIGATÓRIO)
chmod 400 unifaat-ads-key.pem

# Verificar permissões
ls -la unifaat-ads-key.pem
```

**Saída esperada:**
```
-r-- 1 usuario usuario 1704 Jan 15 10:30 unifaat-ads-key.pem
```

### 🔐 Por que chmod 400?

- **400**: Apenas o proprietário pode ler
- **SSH exige** que chaves privadas sejam protegidas
- **Sem isso**: SSH recusará a conexão por segurança
- **Princípio**: Chaves privadas nunca devem ser acessíveis por outros



## Passo 2: Configurando o Firewall (Security Groups)

### O que são Security Groups?

Security Groups funcionam como **firewall virtual** para instâncias EC2. Controlam tráfego de entrada (inbound) e saída (outbound).

### Criando Security Group

**1. Acesse Security Groups:**
- Console EC2 → Security Groups → Create security group

**2. Configurações Básicas:**
- Security group name: `Web-Server-SG`
- Description: `Security group para servidores web - Curso ADS`
- VPC: (deixe o padrão)

**3. Inbound Rules:**

**Regra 1 - SSH:**
- Type: **SSH**
- Protocol: TCP
- Port range: 22
- Source: **My IP** (seu IP atual)
- Description: `SSH access from my IP`

**Regra 2 - HTTP:**
- Type: **HTTP**
- Protocol: TCP
- Port range: 80
- Source: **0.0.0.0/0** (qualquer lugar)
- Description: `HTTP access from anywhere`

**4. Outbound Rules:**
- Deixe o padrão (All traffic - 0.0.0.0/0)

### 🚨 Por que NÃO liberar SSH para 0.0.0.0/0?

**❌ NUNCA FAÇA:**
```
SSH (22) - Source: 0.0.0.0/0
```

**Por que é perigoso:**
- **Ataques de força bruta**: Bots tentarão quebrar senhas
- **Superfície de ataque**: Expõe servidor para o mundo todo
- **Compliance**: Viola políticas de segurança
- **Logs poluídos**: Milhares de tentativas de invasão

**✅ SEMPRE FAÇA:**
```
SSH (22) - Source: Meu IP específico
```

### 🧠 Entendendo CIDR

- **0.0.0.0/0**: Todo o mundo (qualquer IP)
- **192.168.1.100/32**: Apenas um IP específico
- **192.168.1.0/24**: Rede local (256 IPs)
- **My IP**: AWS detecta automaticamente seu IP público

## Passo 4: Conexão e Deploy

### Obtendo IP Público

**No Console:**
- EC2 → Instances → Selecione sua instância
- Copie o **Public IPv4 address**

**Via CLI:**
```bash
# Obter IP público
aws ec2 describe-instances --filters "Name=tag:Name,Values=WebServer-UniFAAT-ADS" --query 'Reservations[*].Instances[*].PublicIpAddress' --output text
```

### Conectando via SSH

```bash
# Conectar na instância (substitua IP_PUBLICO)
ssh -i "unifaat-ads-key.pem" ec2-user@IP_PUBLICO
```

**Exemplo:**
```bash
ssh -i "unifaat-ads-key.pem" ec2-user@54.123.45.67
```

**Na primeira conexão:**
- Aparecerá: `Are you sure you want to continue connecting? (yes/no)`
- Digite: `yes`

### Instalação do Nginx

**Dentro da instância EC2:**

```bash
# Atualizar sistema
sudo dnf update -y

# Instalar Nginx
sudo dnf install nginx -y

# Habilitar e iniciar Nginx
sudo systemctl enable --now nginx

# Verificar status
sudo systemctl status nginx
```

**Saída esperada:**
```
● nginx.service - The nginx HTTP and reverse proxy server
   Loaded: loaded (/usr/lib/systemd/system/nginx.service; enabled; vendor preset: disabled)
   Active: active (running) since Mon 2024-01-15 14:30:45 UTC; 10s ago
```

### 🧠 Diferença: IP Público vs IP Privado

| Aspecto | IP Público | IP Privado |
||||
| **Acesso** | Internet | Apenas VPC |
| **Mudança** | Muda no reboot | Fixo durante vida da instância |
| **Custo** | Pode ser cobrado | Gratuito |
| **Exemplo** | 54.123.45.67 | 172.31.32.45 |
| **Uso** | Acesso externo | Comunicação interna |

## 🧹 Procedimento de Limpeza

### ⚠️ IMPORTANTE: Evitar Custos Desnecessários

**Após completar o lab:**

### Parar a Instância (Temporário)

```bash
# Via CLI
aws ec2 stop-instances --instance-ids i-1234567890abcdef0

# Via Console
# EC2 → Instances → Selecionar instância → Instance state → Stop instance
```

**Características:**
- **Custo**: Para de cobrar compute, mas mantém storage
- **IP Público**: Será perdido
- **Dados**: Preservados no EBS
- **Reinício**: Pode ser reiniciada depois

### Terminar a Instância (Permanente)

```bash
# Via CLI
aws ec2 terminate-instances --instance-ids i-1234567890abcdef0

# Via Console
# EC2 → Instances → Selecionar instância → Instance state → Terminate instance
```

**Características:**
- **Custo**: Para toda cobrança
- **Dados**: **PERDIDOS PERMANENTEMENTE**
- **Recuperação**: **IMPOSSÍVEL**
- **Uso**: Quando não precisa mais da instância

### Limpeza Completa

```bash
# 1. Terminar instância
aws ec2 terminate-instances --instance-ids $(aws ec2 describe-instances --filters "Name=tag:Name,Values=WebServer-UniFAAT-ADS" --query 'Reservations[*].Instances[*].InstanceId' --output text)

# 2. Aguardar terminação
aws ec2 wait instance-terminated --instance-ids INSTANCE_ID

# 3. Deletar Security Group (opcional)
aws ec2 delete-security-group --group-name "Web-Server-SG"

# 4. Deletar Key Pair (opcional)
aws ec2 delete-key-pair --key-name "unifaat-ads-key"
```



## 🆘 Troubleshooting Comum

### "Permission denied (publickey)"

**Problema:** SSH não consegue conectar  
**Soluções:**
```bash
# Verificar permissões da chave
ls -la unifaat-ads-key.pem
# Deve mostrar: -r--

# Corrigir permissões
chmod 400 unifaat-ads-key.pem

# Verificar usuário correto
# Amazon Linux: ec2-user
# Ubuntu: ubuntu
# CentOS: centos
```

### "Connection timed out"

**Problema:** Não consegue conectar na porta 22  
**Soluções:**
- Verificar Security Group (porta 22 liberada para seu IP)
- Confirmar IP público da instância
- Verificar se instância está rodando

### "This site can't be reached"

**Problema:** Navegador não acessa o site  
**Soluções:**
```bash
# Verificar se Nginx está rodando
sudo systemctl status nginx

# Verificar Security Group (porta 80 liberada)
# Confirmar IP público correto
```

### Instância não inicia

**Problema:** Status fica em "pending"  
**Soluções:**
- Aguardar alguns minutos
- Verificar limites da conta (Free Tier)
- Tentar região diferente
