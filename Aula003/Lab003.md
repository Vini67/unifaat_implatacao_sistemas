## Passo 1: Instalação do Plugin Compose

### Verificando se o Plugin já está Instalado

O Docker Compose moderno vem integrado ao Docker Engine. Vamos verificar:

```bash
docker compose version
```

Se o comando funcionar, você já tem o plugin instalado. Caso contrário, vamos instalar:

### Instalando o Plugin Compose

```bash
sudo apt update
sudo apt install docker-compose-plugin -y
```

### Verificando a Instalação

```bash
docker compose version
```

**Saída esperada:**
```
Docker Compose version v2.24.1
```

### Diferença: docker-compose vs docker compose

| Aspecto | docker-compose (v1) | docker compose (v2) |
|---------|-------------------|-------------------|
| **Instalação** | Pacote Python separado | Plugin integrado ao Docker |
| **Comando** | `docker-compose up` | `docker compose up` |
| **Performance** | Mais lento | Mais rápido (escrito em Go) |
| **Manutenção** | Descontinuado | Versão atual e mantida |
| **Compatibilidade** | YAML v2/v3 | YAML v3+ com recursos extras |

> ⚠️ **Importante:** Sempre use `docker compose` (com espaço) - é a versão moderna e mantida.

## Passo 3: Escrevendo o YAML

### Criando o docker-compose.yml

```bash
nano docker-compose.yml
```

**Digite o seguinte conteúdo:**

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    container_name: web-frontend
    ports:
      - "8080:80"
    volumes:
      - ./app:/usr/share/nginx/html
    networks:
      - app-network
    restart: always
    depends_on:
      - app

  app:
    image: httpd:alpine
    container_name: app-backend
    ports:
      - "8081:80"
    networks:
      - app-network
    restart: always
    environment:
      - ENV=development
      - SERVICE_NAME=backend-api

networks:
  app-network:
    driver: bridge

volumes:
  app-data:
    driver: local
```

### Explicação Detalhada de Cada Diretiva

**version: '3.8'**
- Define a versão do formato do Compose file
- Versão 3.8 suporta recursos modernos e é compatível com Docker Swarm

**services:**
- Define todos os containers que compõem sua aplicação
- Cada serviço é um container independente

**image: nginx:alpine**
- Especifica a imagem Docker a ser usada
- Alpine = distribuição Linux minimalista (~5MB)

**container_name: web-frontend**
- Nome personalizado para o container (opcional)
- Facilita identificação e debugging

**ports: - "8080:80"**
- Mapeia porta do host (8080) para porta do container (80)
- Formato: "HOST:CONTAINER"

**volumes: - ./app:/usr/share/nginx/html**
- Monta diretório local dentro do container
- Permite edição de arquivos sem rebuild

**networks: - app-network**
- Conecta o serviço à rede personalizada
- Permite comunicação entre containers por nome

**restart: always**
- Política de reinicialização automática
- Opções: no, always, on-failure, unless-stopped

**depends_on: - app**
- Define ordem de inicialização
- web só inicia depois que app estiver rodando

**environment:**
- Define variáveis de ambiente dentro do container
- Útil para configurações específicas por ambiente

**networks: app-network:**
- Cria rede personalizada para os serviços
- driver: bridge = rede local entre containers

### ⚠️ Cuidados com YAML

```yaml
# ✅ CORRETO - Identação com 2 espaços
services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"

# ❌ ERRADO - Mistura tabs e espaços
services:
	web:
    image: nginx:alpine
	  ports:
      - "8080:80"
```

> **Dica do Professor:** YAML é sensível à identação! Use sempre 2 espaços, nunca tabs. Configure seu editor para mostrar espaços em branco.

## Passo 4: Ciclo de Vida do Compose

### Iniciando o Stack

```bash
docker compose up -d
```

**Flags importantes:**
- `-d`: Detached mode (roda em background)
- `--build`: Força rebuild das imagens
- `--force-recreate`: Recria containers mesmo se não houve mudanças

### Verificando os Serviços

```bash
docker compose ps
```

**Saída esperada:**
```
NAME            IMAGE          COMMAND                  SERVICE   CREATED         STATUS         PORTS
app-backend     httpd:alpine   "httpd-foreground"       app       2 minutes ago   Up 2 minutes   0.0.0.0:8081->80/tcp
web-frontend    nginx:alpine   "/docker-entrypoint.…"   web       2 minutes ago   Up 2 minutes   0.0.0.0:8080->80/tcp
```

### Visualizando Logs

```bash
# Logs de todos os serviços
docker compose logs -f

# Logs de um serviço específico
docker compose logs -f web

# Últimas 50 linhas
docker compose logs --tail=50
```

### Testando os Serviços

```bash
# Testar o frontend (Nginx)
curl http://localhost:8080

# Testar o backend (Apache)
curl http://localhost:8081
```

### Parando e Removendo o Stack

```bash
# Parar serviços (containers permanecem)
docker compose stop

# Iniciar serviços parados
docker compose start

# Parar e remover tudo
docker compose down
```

### Por que `down` é Mais Potente?

| Comando | Containers | Redes | Volumes |
|---------|------------|-------|---------|
| `docker compose stop` | Para | Mantém | Mantém |
| `docker compose down` | Remove | Remove | Mantém* |
| `docker compose down -v` | Remove | Remove | Remove |

*Volumes nomeados são preservados por padrão

## Passo 5: Rede e Comunicação Inter-container

### Entendendo a Rede Interna

Quando você cria serviços no Compose, eles podem se comunicar usando os nomes definidos no YAML como DNS.

### Testando Comunicação Entre Containers

```bash
# Entrar no container web
docker exec -it web-frontend sh
```

**Dentro do container, execute:**

```bash
# Testar conectividade com ping
ping app

# Fazer requisição HTTP para o serviço app
wget -qO- http://app

# Ou usando curl (se disponível)
curl http://app
```

### Verificando a Resolução DNS

```bash
# Ainda dentro do container web
nslookup app
```

**Saída esperada:**
```
Server:    127.0.0.11
Address:   127.0.0.11:53

Name:      app
Address:   172.18.0.3
```

### Inspecionando a Rede

```bash
# Sair do container
exit

# Listar redes Docker
docker network ls

# Inspecionar a rede do projeto
docker network inspect aula03_app-network
```

### Como Funciona o DNS Interno?

1. **Docker cria um DNS server interno** (127.0.0.11)
2. **Cada container recebe um IP** na rede personalizada
3. **Nomes dos serviços** se tornam hostnames válidos
4. **Resolução automática** entre containers na mesma rede

> 💡 **Conceito Fundamental:** Esta é a base dos microsserviços! Serviços se comunicam por nome, não por IP fixo.

## Passo 6: Persistência com Volumes

### Atualizando o docker-compose.yml

Vamos adicionar um volume nomeado para demonstrar persistência:

```bash
nano docker-compose.yml
```

**Adicione ao serviço app:**

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    container_name: web-frontend
    ports:
      - "8080:80"
    volumes:
      - ./app:/usr/share/nginx/html
    networks:
      - app-network
    restart: always
    depends_on:
      - app

  app:
    image: httpd:alpine
    container_name: app-backend
    ports:
      - "8081:80"
    volumes:
      - app-data:/usr/local/apache2/htdocs
      - ./app:/usr/local/apache2/htdocs
    networks:
      - app-network
    restart: always
    environment:
      - ENV=development
      - SERVICE_NAME=backend-api

networks:
  app-network:
    driver: bridge

volumes:
  app-data:
    driver: local
```

### Aplicando as Mudanças

```bash
# Parar o stack atual
docker compose down

# Iniciar com as novas configurações
docker compose up -d
```

### Testando a Persistência

```bash
# Criar arquivo no volume
docker exec -it app-backend sh -c "echo 'Dados persistentes!' > /usr/local/apache2/htdocs/teste.txt"

# Verificar se o arquivo existe
curl http://localhost:8081/teste.txt

# Recriar o container
docker compose down
docker compose up -d

# Verificar se o arquivo ainda existe
curl http://localhost:8081/teste.txt
```

### Tipos de Volume no Compose

```yaml
volumes:
  # Bind mount - mapeia diretório do host
  - ./app:/usr/share/nginx/html
  
  # Volume nomeado - gerenciado pelo Docker
  - app-data:/data
  
  # Volume anônimo - temporário
  - /tmp
  
  # Volume com opções específicas
  - type: bind
    source: ./app
    target: /usr/share/nginx/html
    read_only: true
```

## Troubleshooting Comum

### "Port already in use"

**Problema:** Porta 8080 ou 8081 já está sendo usada  
**Solução:** Alterar as portas no docker-compose.yml

```yaml
ports:
  - "8082:80"  # Usar porta diferente
```

### "YAML parse error"

**Problema:** Erro de sintaxe no arquivo YAML  
**Solução:** Verificar identação e sintaxe

```bash
# Validar sintaxe do YAML
docker compose config
```

### Containers não se comunicam

**Problema:** Ping/curl entre containers falha  
**Solução:** Verificar se estão na mesma rede

```bash
# Verificar redes dos containers
docker inspect web-frontend | grep NetworkMode
docker inspect app-backend | grep NetworkMode
```

### Volume não persiste dados

**Problema:** Dados são perdidos ao recriar containers  
**Solução:** Verificar configuração de volumes

```bash
# Listar volumes
docker volume ls

# Inspecionar volume específico
docker volume inspect aula03_app-data
```

## 🚀 Conceitos Avançados (Bônus)

### Profiles para Diferentes Ambientes

```yaml
services:
  web:
    # ... configuração básica
  
  debug:
    image: nginx:alpine
    profiles: ["debug"]
    ports:
      - "9090:80"
```

```bash
# Rodar apenas serviços básicos
docker compose up -d

# Rodar com profile debug
docker compose --profile debug up -d
```

### Health Checks

```yaml
services:
  web:
    image: nginx:alpine
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

### Scaling Horizontal

```bash
# Escalar serviço para 3 instâncias
docker compose up -d --scale app=3
```
