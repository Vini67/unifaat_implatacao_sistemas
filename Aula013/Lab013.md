## Passo 1: Criando o Canal de Comunicação (SNS)

### O que é Amazon SNS?

**Simple Notification Service** é um serviço de mensageria que permite enviar notificações para múltiplos destinos: e-mail, SMS, aplicações móveis, webhooks, etc.

### Criando o Tópico SNS

**1. Acesse SNS:**
- Console AWS → SNS → Topics → Create topic

**2. Configurações do Tópico:**
- Type: **Standard** (mais recursos, ordem não garantida)
- Name: `Alerta-Sistema-Critico`
- Display name: `Alertas Críticos - UniFAAT ADS`

**3. Configurações Avançadas:**
- Access policy: **Basic** (apenas owner pode publicar)
- Delivery retry policy: **Default**
- Delivery status logging: **Disabled** (economizar custos)

**4. Encryption:**
- **Disabled** (para simplificar o lab)

**5. Create topic**

### Configurando Assinatura (Subscription)

**1. No tópico criado:**
- Clique no tópico `Alerta-Sistema-Critico`
- Aba **Subscriptions** → Create subscription

**2. Configurar Subscription:**
- Topic ARN: (já preenchido)
- Protocol: **Email**
- Endpoint: **SEU-EMAIL-PESSOAL@exemplo.com**

**3. Create subscription**

### ⚠️ OBRIGATÓRIO: Confirmar Assinatura

**1. Verificar e-mail:**
- Acesse sua caixa de entrada
- Procure por: "AWS Notification - Subscription Confirmation"
- **Clique em "Confirm subscription"**

**2. Validar confirmação:**
- Status deve mudar de "Pending confirmation" para "Confirmed"

### Testando o Canal

```bash
# Obter ARN do tópico
TOPIC_ARN=$(aws sns list-topics --query 'Topics[?contains(TopicArn,`Alerta-Sistema-Critico`)].TopicArn' --output text)

# Enviar mensagem de teste
aws sns publish --topic-arn $TOPIC_ARN --message "Teste de conectividade - Sistema de alertas funcionando!" --subject "Teste - Lab 13 UniFAAT"
```

**Resultado esperado:** E-mail de teste recebido em poucos segundos.



## Passo 2: Configurando o Alarme de CPU

### Acessando CloudWatch

**1. Console CloudWatch:**
- Console AWS → CloudWatch → Alarms → Create alarm

### Selecionando Métrica

**2. Select metric:**
- **EC2** → Per-Instance Metrics
- Procure pelo Instance ID da sua EC2
- Selecione métrica: **CPUUtilization**
- Select metric

### Configurando Condições

**3. Specify metric and conditions:**
- Metric name: `CPUUtilization`
- Statistic: **Average**
- Period: **1 minute**

**4. Conditions:**
- Threshold type: **Static**
- Whenever CPUUtilization is: **Greater**
- than: **60**

### Configurando Ações

**5. Configure actions:**
- Alarm state trigger: **In alarm**
- Select an SNS topic: **Select an existing SNS topic**
- SNS topic: `Alerta-Sistema-Critico`

**6. Add notification (opcional):**
- OK state trigger: **In alarm** → **OK**
- SNS topic: `Alerta-Sistema-Critico`

### Nomeando o Alarme

**7. Add name and description:**
- Alarm name: `CPU-Alta-EC2-UniFAAT`
- Alarm description: `Alarme de CPU alta na instância EC2 - Curso ADS`

**8. Create alarm**

### 🧠 Estados do Alarme

| Estado | Cor | Significado | Ação |
|--|--|-||
| **OK** | 🟢 Verde | Métrica dentro do normal | Nenhuma |
| **ALARM** | 🔴 Vermelho | Threshold ultrapassado | Enviar notificação |
| **INSUFFICIENT_DATA** | 🟡 Cinza | Dados insuficientes | Aguardar dados |



## Passo 3: Teste de Estresse (Simulando o Incidente)

### Conectando na Instância EC2

```bash
# Obter IP público da EC2
EC2_IP=$(aws ec2 describe-instances --filters "Name=instance-state-name,Values=running" --query 'Reservations[0].Instances[0].PublicIpAddress' --output text)

# Conectar via SSH
ssh -i "unifaat-ads-key.pem" ec2-user@$EC2_IP
```

### Instalando Ferramenta de Estresse

**Dentro da instância EC2:**

```bash
# Instalar stress tool
sudo dnf install stress -y

# Verificar instalação
stress --version

# Verificar número de CPUs disponíveis
nproc
```

### 🎯 Desafio: Provocando o Incidente

**1. Abrir CloudWatch em outra aba:**
- CloudWatch → Alarms → CPU-Alta-EC2-UniFAAT
- Manter aba aberta para acompanhar

**2. Executar teste de estresse:**

```bash
# Gerar carga de CPU por 5 minutos
stress --cpu 1 --timeout 300

# Para instâncias com múltiplas CPUs, ajustar:
# stress --cpu $(nproc) --timeout 300
```

**3. Monitorar em tempo real:**
- **CloudWatch**: Gráfico deve subir
- **Terminal**: Comando stress rodando
- **E-mail**: Aguardar notificação de alarme

### Comandos de Monitoramento

**Em outro terminal SSH:**

```bash
# Monitorar CPU em tempo real
top

# Ou usar htop se disponível
sudo dnf install htop -y
htop

# Verificar carga do sistema
uptime

# Monitorar por linha de comando
watch -n 1 'cat /proc/loadavg'
```

### Resultados Esperados

**Sequência de eventos:**
1. **0-60s**: CPU sobe gradualmente
2. **60-120s**: CPU > 60%, alarme dispara
3. **120s**: E-mail de alerta recebido
4. **300s**: Stress termina, CPU normaliza
5. **360s**: Alarme volta ao estado OK

```bash
for i in {1..10}; do
  curl http://localhost/
```

### Configuração Avançada (CloudWatch Agent)

**Para centralização automática (conceitual):**

```bash
# Instalar CloudWatch Agent (não executar - apenas demonstração)
sudo dnf install amazon-cloudwatch-agent -y

# Configuração seria via arquivo JSON:
# /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json
```

**Exemplo de configuração:**
```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/nginx/access.log",
            "log_group_name": "/aws/ec2/nginx/access",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

### 🧠 Benefícios da Centralização

**Problemas do `tail -f`:**
- Acesso apenas via SSH
- Não persiste histórico
- Difícil de pesquisar
- Sem alertas automáticos

**Vantagens do CloudWatch Logs:**
- Acesso via console web
- Retenção configurável
- Busca e filtros avançados
- Integração com alarmes
- Análise com CloudWatch Insights

## 🧠 Conceitos de Monitoramento

### Status Check: Hardware vs Software

| Tipo | O que Monitora | Falha Indica | Ação Típica |
||-|--|-|
| **System Status** | Hardware AWS | Problema físico | AWS resolve automaticamente |
| **Instance Status** | SO e aplicação | Problema de software | Reiniciar instância |

### Métricas Essenciais

**CPU:**
- **CPUUtilization**: % de uso da CPU
- **Threshold típico**: > 80% por 5 minutos

**Memória:**
- **Requer CloudWatch Agent**
- **Threshold típico**: > 85%

**Disco:**
- **DiskSpaceUtilization**: % de uso do disco
- **Threshold típico**: > 90%

**Rede:**
- **NetworkIn/Out**: Bytes transferidos
- **NetworkPacketsIn/Out**: Pacotes transferidos

## 📚 Comandos de Referência Rápida

```bash
# SNS
aws sns list-topics                              # Listar tópicos
aws sns publish --topic-arn ARN --message MSG    # Enviar mensagem
aws sns delete-topic --topic-arn ARN             # Deletar tópico

# CloudWatch Alarms
aws cloudwatch describe-alarms                   # Listar alarmes

# CloudWatch Metrics
aws cloudwatch list-metrics                      # Listar métricas
aws cloudwatch get-metric-statistics             # Obter dados de métrica

# CloudWatch Logs
```

## 🎓 Conexão com SRE e DevOps

**Princípios de SRE:**

- **SLI (Service Level Indicators)**: Métricas que importam
- **SLO (Service Level Objectives)**: Metas de performance
- **Error Budget**: Tolerância a falhas
- **Toil Reduction**: Automação de tarefas repetitivas

**Alerting Best Practices:**

- **Actionable**: Todo alerta deve ter ação clara
- **Proportional**: Severidade adequada ao impacto
- **Rare**: Evitar fadiga de alertas
- **Timely**: Detectar problemas rapidamente

**Observability Pillars:**

- **Metrics**: Dados numéricos agregados
- **Logs**: Eventos detalhados
- **Traces**: Fluxo de requisições
