## Passo 1: Entendendo o Healthcheck

### Criando a Estrutura do Projeto

```bash
mkdir -p ~/ImplantacaoSistemas/Aula05
cd ~/ImplantacaoSistemas/Aula05
```

### Criando o docker-compose.yml com Healthcheck

```bash
nano docker-compose.yml
```

**Digite o seguinte conteúdo:**

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    container_name: web-app
    ports:
      - "8080:80"
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    restart: always

networks:
  app-network:
    driver: bridge
```

### Explicação dos Parâmetros de Healthcheck

**test: ["CMD", "curl", "-f", "http://localhost"]**
- `CMD`: Executa comando dentro do container
- `curl -f`: Faz requisição HTTP e falha se status não for 2xx
- `http://localhost`: Endpoint a ser testado

**interval: 30s**
- Intervalo entre execuções do teste
- Padrão: 30 segundos

**timeout: 10s**
- Tempo máximo para o comando executar
- Se exceder, considera falha

**retries: 3**
- Número de tentativas consecutivas antes de marcar como unhealthy
- Evita falsos positivos

**start_period: 40s**
- **Grace Period**: Tempo de carência após inicialização
- Falhas durante este período não contam para retries
- Importante para aplicações que demoram a iniciar

### Instalando curl no Container

Como nginx:alpine não tem curl por padrão, vamos criar um Dockerfile:

```bash
nano Dockerfile
```

**Digite o seguinte conteúdo:**

```dockerfile
FROM nginx:alpine

# Instalar curl para healthcheck
RUN apk add --no-cache curl

# Copiar configuração padrão (opcional)
COPY nginx.conf /etc/nginx/nginx.conf

# Expor porta
EXPOSE 80
```

### Atualizando o docker-compose.yml

## Passo 2: Dependência Inteligente

### Adicionando Banco de Dados com Healthcheck

```bash
nano docker-compose.yml
```

**Substitua o conteúdo por:**

```yaml
version: '3.8'

services:
  db:
    image: postgres:alpine
    container_name: postgres-db
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: apppass
    networks:
      - app-network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: always
    volumes:
      - db-data:/var/lib/postgresql/data

  web:
    build: .
    container_name: web-app
    ports:
      - "8080:80"
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    depends_on:
      db:
        condition: service_healthy
    restart: always

networks:
  app-network:
    driver: bridge

volumes:
  db-data:
    driver: local
```

### Dependência Inteligente vs. Simples

**depends_on simples:**
```yaml
depends_on:
  - db
```
- Apenas garante ordem de inicialização
- Não verifica se o serviço está funcionando
- Pode causar falhas se DB não estiver pronto

**depends_on com condition:**
```yaml
depends_on:
  db:
    condition: service_healthy
```
- Aguarda até o serviço estar **healthy**
- Garante que dependências estejam funcionais
- Evita falhas de conexão durante inicialização

### Testando a Dependência

```bash
# Construir e iniciar os serviços
docker compose up -d --build

# Observar a ordem de inicialização
docker compose logs -f
```

**O que observar:**
1. DB inicia primeiro
2. DB executa healthchecks até ficar healthy
3. Web só inicia após DB estar healthy

## Passo 4: Automação de Manutenção

### Criando o Script de Limpeza

```bash
nano cleanup.sh
```

**Digite o seguinte conteúdo:**

```bash
#!/bin/bash

echo "🧹 Iniciando limpeza do sistema Docker..."
echo "⚠️  ATENÇÃO: Este script removerá recursos não utilizados!"
echo

# Confirmar ação
read -p "Deseja continuar? (y/N): " -n 1 -r
echo
if [[ ! $REPLY =~ ^[Yy]$ ]]; then
    echo "❌ Operação cancelada."
    exit 1
fi

echo "🗑️  Removendo containers parados..."
docker container prune -f

echo "🗑️  Removendo imagens não utilizadas..."
docker image prune -f

echo "🗑️  Removendo redes não utilizadas..."
docker network prune -f

echo "🗑️  Removendo volumes não utilizados..."
echo "⚠️  CUIDADO: Volumes contêm dados persistentes!"
read -p "Remover volumes não utilizados? (y/N): " -n 1 -r
echo
if [[ $REPLY =~ ^[Yy]$ ]]; then
    docker volume prune -f
    echo "✅ Volumes removidos."
else
    echo "⏭️  Volumes preservados."
fi

echo "🗑️  Removendo cache de build..."
docker builder prune -f

echo
echo "📊 Espaço liberado:"
docker system df

echo
echo "✅ Limpeza concluída!"
```

### Tornando o Script Executável

```bash
chmod +x cleanup.sh
```

### Riscos e Benefícios da Limpeza

**Benefícios:**
- **Libera espaço em disco**: Remove imagens, containers e redes não utilizados
- **Melhora performance**: Menos objetos para gerenciar
- **Organização**: Mantém ambiente limpo

**Riscos:**
- **Perda de dados**: Volumes podem conter dados importantes
- **Rebuild necessário**: Imagens removidas precisarão ser baixadas novamente
- **Downtime**: Containers em uso podem ser afetados

**Boas práticas:**
- Sempre confirmar antes de executar
- Fazer backup de volumes importantes
- Executar em horários de baixo uso

### Monitorando Eventos em Tempo Real

```bash
# Monitorar todos os eventos
docker events

# Filtrar eventos específicos
docker events --filter event=health_status

# Filtrar por container específico
docker events --filter container=web-app
```

### Testando Monitoramento de Eventos

**Em um terminal, execute:**
```bash
docker events --filter event=health_status --filter event=die --filter event=start
```

**Em outro terminal, execute:**
```bash
# Parar e iniciar container
docker compose stop web
docker compose start web

# Forçar falha de healthcheck (se ainda estiver configurado)
docker compose restart web
```

### Eventos Importantes para Monitorar

**Estados de Saúde:**
- `health_status: healthy` - Container passou no teste
- `health_status: unhealthy` - Container falhou no teste

**Estados de Ciclo de Vida:**
- `start` - Container iniciado
- `die` - Container parou
- `restart` - Container reiniciado

**Exemplo de saída:**
```

2024-01-15T10:31:15.987654321Z container health_status: unhealthy (id: abc123...)
2024-01-15T10:31:45.456789123Z container die (id: abc123...)
2024-01-15T10:31:46.789123456Z container start (id: def456...)
```



## Conceito de Grace Period

### Por que Grace Period é Importante?

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost"]
  start_period: 60s  # Grace period de 60 segundos
  retries: 3
```

**Sem Grace Period:**
- Aplicação demora 45s para inicializar
- Healthcheck falha 3 vezes nos primeiros 45s
- Container marcado como unhealthy prematuramente

**Com Grace Period:**
- Falhas nos primeiros 60s não contam
- Aplicação tem tempo para inicializar completamente
- Healthcheck só conta após período de carência

### Configurações por Tipo de Aplicação

**Aplicações Web Simples:**
```yaml
start_period: 30s
interval: 30s
timeout: 10s
retries: 3
```

**Bancos de Dados:**
```yaml
start_period: 60s
interval: 10s
timeout: 5s
retries: 5
```

**Aplicações Java/Spring:**
```yaml
start_period: 120s  # JVM demora para inicializar
interval: 30s
timeout: 15s
retries: 3
```

## Troubleshooting Comum

### Container sempre unhealthy

**Problema:** Healthcheck sempre falha  
**Solução:** Verificar se comando funciona manualmente

```bash
# Testar comando dentro do container
docker exec web-app curl -f http://localhost
```

### Dependência não funciona

**Problema:** Web inicia mesmo com DB unhealthy  
**Solução:** Verificar sintaxe do depends_on

```yaml
# ✅ CORRETO
depends_on:
  db:
    condition: service_healthy

# ❌ ERRADO
depends_on:
  - db:
      condition: service_healthy
```

### Grace period muito curto

**Problema:** Container marcado como unhealthy muito cedo  
**Solução:** Aumentar start_period

```yaml
healthcheck:
  start_period: 120s  # Aumentar para aplicações lentas
```

## Conceitos Avançados (Bônus)

### Healthcheck com Script Personalizado

```dockerfile
FROM nginx:alpine
RUN apk add --no-cache curl
COPY healthcheck.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/healthcheck.sh
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD /usr/local/bin/healthcheck.sh
```

### Múltiplos Testes de Saúde

```bash
#!/bin/bash
# healthcheck.sh
curl -f http://localhost/health && \
curl -f http://localhost/api/status && \
test -f /app/ready.flag
```

### Healthcheck com Métricas

```yaml
healthcheck:
  test: ["CMD-SHELL", "curl -f http://localhost/metrics | grep 'app_ready 1'"]
```
